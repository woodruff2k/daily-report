## 1\. 개요

본 문서는 영업 일일 보고 시스템의 기능·권한·비기능 요구사항이 정상 동작하는지 검증하기 위한 테스트 케이스를 정의한다. 각 테스트 케이스는 기능 요구사항(FR), API 엔드포인트, 화면(SCR)과 추적 가능하도록 연결한다.

### 1.1 테스트 범위

- 인증, 일일보고(작성·제출·조회), 방문기록·과제·계획, 댓글, 고객/영업 마스터  
- 권한(인가) 검증, 입력 유효성, 주요 비기능(이력·무결성) 검증

### 1.2 테스트 유형

| 유형 | 설명 |
| :---- | :---- |
| 단위 | 개별 함수/유효성 로직 검증 |
| 통합(API) | API 명세서 기준 엔드포인트 단위 검증 |
| 시스템(E2E) | 화면 흐름 기준 사용자 시나리오 검증 |
| 권한/보안 | 인가, IDOR, 입력 검증, 개인정보 노출 검증 |

### 1.3 우선순위 정의

- **High**: 핵심 기능/권한, 실패 시 서비스 사용 불가  
- **Medium**: 보조 기능/예외 처리  
- **Low**: UI 표기·정렬 등

### 1.4 테스트 데이터 원칙

- 실제 고객/직원/운영 데이터를 사용하지 않고 **합성(더미) 데이터**만 사용한다.  
- 계정/토큰 등 자격증명은 테스트 전용 더미 값을 사용한다.

## 3\. 합격 기준 및 결함 관리

### 3.1 합격 기준

- High 우선순위 케이스 **100% Pass**  
- Medium 이상 전체 **95% 이상 Pass**, 잔여 결함은 위험도 평가 후 릴리스 판단

### 3.2 결함 심각도

| 등급 | 정의 |
| :---- | :---- |
| Critical | 서비스 중단, 권한 우회, 개인정보 노출 |
| Major | 핵심 기능 오류, 우회책 없음 |
| Minor | 보조 기능 오류, 우회 가능 |
| Trivial | UI/표기 등 경미 |

### 3.3 결함 리포트 항목

TC ID, 재현 절차, 입력 데이터(합성), 기대 결과, 실제 결과, 심각도, 첨부(로그/캡처), 담당자.
