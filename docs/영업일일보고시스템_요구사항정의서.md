## 1\. 시스템 개요

영업사원이 매일 방문 활동을 보고하고, 과제/상담(Problem)과 익일 계획(Plan)을 작성하면 상급자가 댓글로 피드백을 남기는 일일 보고 시스템이다. 고객·영업사원은 마스터로 관리한다.

### 1.1 목적

- 영업 활동(방문/상담)의 일일 기록 및 가시화  
- 과제·계획에 대한 상급자 피드백 채널 제공  
- 고객·영업 마스터 기반의 데이터 일관성 확보

### 1.2 사용자(Actor)

| 사용자 | 설명 | 주요 권한 |
| :---- | :---- | :---- |
| 영업사원 | 일일보고 작성자 | 본인 보고 작성·조회 |
| 상급자 | 팀원 보고 검토자 | 팀원 보고 조회, 댓글 작성 |
| 관리자 | 마스터 관리자 | 영업 마스터 관리 |

## 2\. 기능 요구사항 (Functional Requirements)

| ID | 구분 | 요구사항 |
| :---- | :---- | :---- |
| FR-01 | 마스터 | 고객 마스터를 등록·조회·수정·비활성화할 수 있다. |
| FR-02 | 마스터 | 영업(사원) 마스터를 등록·조회·수정·비활성화할 수 있다. 상급자(관리자) 관계를 가진다. |
| FR-03 | 일일보고 | 영업사원은 하루에 하나의 일일보고를 작성한다. (사원 \+ 보고일자 단위로 유일) |
| FR-04 | 방문기록 | 하나의 일일보고에 방문 고객·방문 내용을 여러 행으로 추가/수정/삭제할 수 있다. |
| FR-05 | 방문기록 | 각 방문기록은 고객 마스터의 고객을 참조한다. |
| FR-06 | 과제/상담 | 현재의 과제·상담(Problem)을 여러 건 작성할 수 있다. |
| FR-07 | 계획 | 내일 할 일(Plan)을 여러 건 작성할 수 있다. |
| FR-08 | 결재/제출 | 일일보고는 작성중(Draft) → 제출(Submitted) 상태로 전환된다. |
| FR-09 | 댓글 | 상급자는 제출된 일일보고에 댓글로 의견을 남길 수 있다. (대댓글 가능) |
| FR-10 | 조회 | 기간별·사원별·고객별로 보고 내용을 조회·검색할 수 있다. |

## 3\. 비기능 요구사항 (Non-functional Requirements)

| ID | 구분 | 요구사항 |
| :---- | :---- | :---- |
| NFR-01 | 권한 | 영업사원은 본인 보고만 작성, 상급자는 소속 팀원 보고 조회 및 댓글 작성. 모든 권한은 서버 측에서 검증한다. |
| NFR-02 | 이력 | 보고/방문기록의 생성·수정 일시 및 작성자를 기록한다. |
| NFR-03 | 무결성 | 마스터(고객/영업)는 물리 삭제 대신 상태값으로 비활성화하여 과거 보고와의 참조를 유지한다. |
| NFR-04 | 개인정보 | 고객 연락처·이메일, 직원 사번·이메일 등 개인정보의 노출(응답·로그)을 최소화한다. |

## 4\. 데이터 요구사항

### 4.1 핵심 엔터티

| 엔터티 | 설명 |
| :---- | :---- |
| SALES\_REP | 영업 마스터. 자기참조로 상급자(manager) 관계를 표현 |
| CUSTOMER | 고객 마스터. 담당 영업사원을 참조 |
| DAILY\_REPORT | 일일보고. 영업사원 1명 × 1일 \= 1건 |
| VISIT\_RECORD | 방문기록. 일일보고에 N건, 고객을 참조 |
| REPORT\_PROBLEM | 과제/상담(Problem). 일일보고에 N건 |
| REPORT\_PLAN | 내일 할 일(Plan). 일일보고에 N건 |
| REPORT\_COMMENT | 댓글. 일일보고에 N건, 작성자(상급자) 참조, 대댓글 가능 |

### 4.2 주요 제약

- `DAILY_REPORT`는 (`rep_id`, `report_date`) 조합으로 유일하다.  
- 방문기록은 일일보고당 최소 1건 이상이어야 제출할 수 있다.  
- 마스터 참조(담당 영업, 방문 고객)는 비활성화된 항목도 과거 데이터 무결성을 위해 유지한다.

상세 엔터티 속성 및 관계는 별도 문서(ER 다이어그램)를 참조한다.

## 6\. 공통 정책

- **상태 전이**: 일일보고 DRAFT → SUBMITTED. 제출 후 수정 정책은 운영 기준에 따른다.  
- **마스터 비활성화**: 물리 삭제 대신 상태값 비활성화로 참조 무결성을 유지한다.  
- **권한 검증**: 모든 조회·작성·댓글 권한은 서버 측에서 검증한다. (화면 비표시는 접근통제가 아님)  
- **이력 관리**: 생성·수정 일시 및 작성자를 자동 기록한다.

## 7\. 향후 확장(검토 항목)

- 댓글 대상을 보고서 단위 → 과제(Problem)/계획(Plan) 단위로 세분화 (`target_type` \+ `target_id` 다형성)  
- 제출 후 수정/재상신(반려) 워크플로  
- 결재(승인) 단계 추가
